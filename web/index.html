A from-scratch C++/Vulkan implementation of GPU mesh shader-based procedural resurfacing — generating complex geometry entirely on-chip at real-time frame rates.

Overview

What is Gravel?

Gravel is a real-time procedural resurfacing framework built on Vulkan's mesh shader pipeline. It takes a base mesh and dynamically generates new geometric surfaces on every face — including parametric shapes, B-spline control cages, and procedural pebbles — without relying on traditional vertex input assembly.

All geometry originates from storage buffers and is synthesized entirely on-chip inside the task and mesh shader stages. This allows for hardware-accelerated amplification of geometry with adaptive LOD, frustum culling, and back-face culling all happening at the GPU level.

Based on: "Real-time procedural resurfacing using GPU mesh shader" — Raad et al., Computer Graphics Forum 44(2), Eurographics 2025.

Project Biography

Chapter 01
The Problem: Static Geometry in a Dynamic World

Traditional rendering pipelines store every vertex of every surface in memory before a single pixel is drawn. For complex scenes — a gravel road, a cobblestone wall, a field of pebbles — that means millions of pre-computed vertices eating GPU memory and bandwidth.

Procedural resurfacing flips the script. Instead of storing geometry, we generate it on-the-fly. Start with a simple base mesh — like a cube — and let the GPU synthesize detailed surfaces on every face at render time, entirely on-chip.

Based on: "Real-time procedural resurfacing using GPU mesh shader" — Raad et al., Eurographics 2025

Chapter 03
Mesh Amplification: From 6 Faces to Thousands

The mesh shader pipeline's superpower is amplification — the ability to take a single input primitive and emit many output primitives. Each face of the base mesh is subdivided, and for every sub-face, a new geometric element is generated.

At moderate subdivision, each cube face spawns dozens of colored patches. Crank the subdivision higher and the cube becomes densely tessellated — all dispatched as hardware workgroups with zero vertex buffer overhead. The per-face coloring shows exactly which sub-patch belongs to which original face.

Workgroup dispatch is hardware-aware: the task shader calculates how many mesh shader workgroups to emit based on visible geometry complexity.

Chapter 04
The Three-Stage Pipeline

Gravel replaces the traditional vertex pipeline with Vulkan's mesh shader pipeline, built on the VK_EXT_mesh_shader extension.

The task shader runs first — performing frustum culling, back-face culling, and LOD selection. The task ID visualization shows how workgroups are distributed across the mesh. The element type map shows per-face surface type selection — each color represents a different procedural element assigned to that face.

The mesh shader then reads base mesh data from storage buffers and emits fully-formed triangles. The fragment shader handles material shading and lighting.

Chapter 05
Parametric Surfaces

The first family of surfaces computed on-chip are analytical parametric shapes — defined purely by mathematical equations evaluated in the mesh shader.

Spheres and hemispheres emerge from each face of the base mesh, generated entirely from parametric equations. The parametric surface library extends further: torus shapes are computed by sweeping a circle along a circular path, cylinders produce a riveted, industrial texture, and cones create a studded, spiky surface. Each shape is defined by a compact set of parameters — no geometry is stored, only equations.

Supported shapes: torus, sphere, cone, cylinder, hemisphere, Möbius strip, Klein bottle, hyperbolic paraboloid, helicoid, and egg.

Chapter 06
B-Spline Cage Pebbles

Not all procedural elements come from analytical equations. B-spline cage pebbles use a 4×4 control-point cage constructed on-the-fly in the mesh shader and evaluated as a bicubic tensor-product surface — producing smooth, organic stone shapes entirely on-chip.

Each face of the base mesh spawns a pebble by building a 16-vertex control cage in shared memory. The cage is organized into four rows: the base face boundary, an inner fill ring, a rounded extrusion ring, and a fully extruded top. The mesh shader uses the uniform cubic B-spline basis matrix to evaluate the surface — first along rows (u direction), then interpolating those results along columns (v direction). Because cubic B-splines approximate rather than interpolate their control points, an interpolation compensation step adjusts vertex positions so the surface passes through the actual face vertices.

Per-face variation comes from a PCG random number generator seeded by face ID, randomizing extrusion height and proportion. On top of the B-spline surface, 3D Perlin noise with analytical gradients displaces each vertex along its surface normal, perturbing both position and shading normal simultaneously. The noise is sampled in world space to avoid seams at patch boundaries, producing natural, irregular pebble shapes that never repeat.

Tessellation is a power-of-two grid (up to 9×9 = 81 vertices per workgroup). For higher subdivision, hierarchical multi-workgroup tiling kicks in — each sub-workgroup evaluates a 9×9 region of the surface with edge stitching to prevent T-junctions.

Chapter 07
Dragon Scales

Unlike the analytically defined parametric shapes, dragon scales are driven by an artist-authored 11×11 B-spline control cage (121 control points) loaded from an OBJ file into a GPU SSBO at startup — zero per-frame CPU overhead. The mesh shader evaluates this as a bicubic tensor-product surface, mapping UV patch indices to 4×4 subsets of the control grid and computing position via the uniform cubic B-spline basis matrix.

Normals are computed by central finite differences — offsetting UV by a small epsilon in each direction and taking the cross product of the resulting tangent vectors. The evaluated geometry is normalized into unit space using a precomputed AABB, then axis-remapped so the scale's curvature aligns outward along the face normal via offsetVertex().

Scale overlap is a natural result of sizing each element by sqrt(faceArea) × userScaling — scaling elements larger than their host face causes adjacent scales to layer over each other. To break visual uniformity, the task shader applies Rodrigues' rotation formula to the edge tangent around the face normal, using a golden-ratio hash (fract(faceID × 0.618...)) to distribute rotation angles with low discrepancy. The result is a dense, shingled surface where no two adjacent scales face exactly the same direction.

The scale LUT is sorted into V-major grid order by UV coordinate at load time, uploaded as a vec4 SSBO at binding 6, and shared across all mesh and task shader invocations.

Chapter 08
Procedural Fur

Parametric hair pushes cone aspect ratios to the extreme — thousands of thin strands generated entirely on-chip, reading as procedural fur. Each strand follows the surface normal of its parent face, with slight randomization in length and direction to produce a natural, organic look.

The technique scales from character bodies to arbitrary meshes. The dragon hair animation shows fur reacting to skeletal motion in real time, while the hairy man demonstrates full-body fur coverage — all generated per-frame in the mesh shader with zero pre-authored strand data.

Chapter 09
Screen-Space Level of Detail

Generating thousands of high-tessellation elements per frame is expensive. The screen-space LOD system solves this by adapting each element's geometric detail based on its projected pixel coverage, computed in the task shader before any geometry is emitted.

The comparison is striking: at low geometry, each procedural element uses minimal triangles — fast to render but visibly faceted. At high geometry, the same elements are densely tessellated and smooth. The LOD system transitions between these levels per-element, so nearby faces run at high detail while distant ones stay cheap.

LOD transitions happen per-element, not per-object — nearby faces of the same mesh can run at different detail levels simultaneously.

Chapter 10
GRWM: Geometry Resampling and Weighting for Meshes

GRWM is a standalone CUDA-based GPU preprocessing pipeline that analyses a mesh before rendering and produces three binary buffers consumed by Gravel at load time: per-vertex mean curvature (via cotangent Laplacian and cuSPARSE), per-face feature edge flags (via dihedral angle detection), and priority-sorted slot grids for multi-element spawning per face.

At render time, the task shader uses these buffers for curvature-aware density modulation — scaling element size based on local surface curvature so high-curvature regions get denser, more detailed coverage. Feature edge enforcement boosts elements on detected feature edges while shrinking non-feature faces, preserving important silhouette detail. Both effects blend with a global intensity slider.

Slot placement enables multiple procedural elements per face. Each face gets up to 64 priority-sorted UV positions generated via Murmur3 hashing, with priority scores combining distance-from-center, local curvature, and a jitter term. The task shader dispatches compound indices encoding face ID and slot index, interpolating position and normal at each slot's barycentric coordinates. CUB segmented sort ensures the highest-priority slots are always spawned first.

GRWM runs as a background process — the render loop continues while preprocessing completes. On finish, buffers are hot-loaded into GPU SSBOs the next frame with zero pipeline stalls.

Chapter 11
Physically-Based Rendering

Procedural geometry only looks convincing under proper lighting. Gravel implements a Cook-Torrance microfacet BRDF shared across all four rendering pipelines (parametric, pebble, base mesh, and benchmark), centralizing PBR into a single shader library (shading.glsl).

The specular lobe uses the GGX/Trowbridge-Reitz normal distribution function for microfacet highlights, Smith-Schlick geometry attenuation for masking and shadowing between microfacets, and the Schlick Fresnel approximation for angle-dependent reflectance. F0 is computed as mix(dielectricF0, albedo, metallic) — dielectrics get a fixed base reflectance (typically 0.04), while metals reflect their own albedo color. Energy conservation ensures diffuse contribution is scaled by (1 - F) × (1 - metallic), with Lambertian diffuse divided by π.

Environment reflections use either an equirectangular HDR map (with a cheap roughness-based blur that blends the sharp sample toward a 6-axis average) or a fake sky/ground fallback that blends sky and ground colors via smoothstep on the reflection direction's Y component. All specular values are brought into displayable range through ACES filmic tone mapping in the fragment shader before writing to the sRGB swap chain — no HDR framebuffer required.

The Cook-Torrance BRDF adds only ~5-10 ALU ops per fragment over the previous Blinn-Phong — negligible in a mesh shader pipeline where the bottleneck is vertex generation, not shading.

Chapter 13
Environment Reflections

Realistic reflections ground procedural geometry in a believable world. Gravel's PBR pipeline supports image-based lighting (IBL) using HDR environment maps — the same technique used in film and AAA games to capture real-world lighting.

Each environment map produces dramatically different reflection characteristics. An open field yields soft, warm reflections. A studio setup gives controlled, even highlights. A house interior produces intimate, warm-toned bounces. A city skyline creates sharp, high-contrast reflections on metallic surfaces.

Environment maps are sampled per-fragment using the reflected view vector, with roughness controlling mip-level selection for glossy-to-rough material transitions.

Chapter 14
Subdivision vs. Reflection Quality

Subdivision level directly impacts how faithfully environment reflections are captured on procedural surfaces. At low subdivision, reflections appear faceted — the surface normals change abruptly between triangles, producing blocky highlights.

At medium subdivision, the reflection smooths out noticeably. Switching between large and small spheres demonstrates how element scale interacts with reflection quality — smaller elements pack more surface variation per pixel, requiring higher subdivision to resolve clean reflections.

The adaptive LOD system automatically increases subdivision for close-up elements where faceted reflections would be most visible.

Chapter 15
Anti-Aliasing

Procedural geometry with high-frequency detail is especially prone to aliasing artifacts. Gravel tackles this with a multi-layered anti-aliasing strategy, each method targeting a different source of visual noise.

MSAA (4x multisample) handles geometric edge aliasing at the rasterization level. Alpha-to-coverage and coverage fading replace hard on/off element visibility with smooth sub-pixel fade-out — the task shader computes screen-space element size and outputs a smooth alpha, which MSAA hardware resolves into partial coverage instead of flickering dithered discards.

Specular anti-aliasing eliminates bright firefly artifacts from sub-pixel normal variation using projected-space NDF filtering (Tokuyoshi & Kaplanyan 2021). Screen-space normal derivatives widen the GGX roughness parameter, and a procedural extension measures angular divergence between the interpolated and flat face normals to capture cross-element variation that per-triangle derivatives miss.

Proxy shading is the final layer: when elements shrink below a threshold, geometry emission is skipped entirely and replaced with analytically-derived shading parameters — aggregate roughness, mean normal tilt, self-shadow scale, and coverage fraction — precomputed from the parametric surface. The result looks like the geometry because it is derived from the geometry, just without emitting any triangles.
